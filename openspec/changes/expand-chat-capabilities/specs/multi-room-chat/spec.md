## ADDED Requirements

### Requirement: 참여 채팅방 목록
시스템은 인증 사용자가 참여 중인 모든 채팅방을 MUST 조회하고 선택할 수 있게 해야 한다.

#### Scenario: 참여 중인 방 조회
- **WHEN** 인증 사용자가 채팅방 목록을 연다
- **THEN** 시스템은 사용자의 `room_members`에 연결된 방만 목록에 표시한다

#### Scenario: 채팅방이 없는 사용자
- **WHEN** 사용자가 참여 중인 방이 없다
- **THEN** 시스템은 오류 대신 빈 상태와 방 생성 또는 참여 동작을 표시한다

### Requirement: 그룹 채팅방 생성
시스템은 유효한 이름으로 그룹 채팅방을 생성하고 생성자를 관리자 멤버로 같은 트랜잭션에서 MUST 등록해야 한다.

#### Scenario: 유효한 방 생성
- **WHEN** 인증 사용자가 앞뒤 공백을 제외한 1자 이상 100자 이하의 방 이름을 제출한다
- **THEN** 시스템은 `group` 방과 생성자의 `admin` 멤버 행을 원자적으로 생성하고 해당 방을 선택한다

#### Scenario: 방 생성 응답 유실 후 재시도
- **WHEN** 클라이언트가 동일 요청 ID로 방 생성을 재시도한다
- **THEN** 시스템은 방을 추가 생성하지 않고 최초 생성된 방을 반환한다

#### Scenario: 유효하지 않은 방 이름
- **WHEN** 비어 있거나 100자를 초과한 방 이름을 제출한다
- **THEN** 시스템은 DB 요청 없이 입력 오류를 표시한다

### Requirement: 그룹 채팅방 참여와 나가기
시스템은 인증 사용자가 허용된 그룹방에 참여하고 자신이 참여한 그룹방에서 나갈 수 있게 MUST 해야 한다.

#### Scenario: 그룹방 참여
- **WHEN** 사용자가 참여 가능한 그룹방에서 참여를 선택한다
- **THEN** 시스템은 본인의 `member` 행을 멱등하게 생성하고 방 접근을 허용한다

#### Scenario: 그룹방 나가기
- **WHEN** 일반 멤버가 참여 중인 그룹방에서 나가기를 확인한다
- **THEN** 시스템은 본인의 멤버 행을 삭제하고 해당 방의 후속 메시지 접근을 차단한다

#### Scenario: 기존 공용방 나가기 시도
- **WHEN** 사용자가 기본 공용방에서 나가기를 시도한다
- **THEN** 시스템은 공용방 참여 정책에 따라 나가기를 허용하지 않는다

### Requirement: 방별 메시지 격리
시스템은 현재 선택한 방의 메시지만 조회·전송·실시간 수신하도록 MUST 격리해야 한다.

#### Scenario: 다른 방으로 전환
- **WHEN** 사용자가 현재 방에서 다른 참여 방을 선택한다
- **THEN** 시스템은 이전 방 구독을 해제하고 새 방의 최근 메시지 조회와 Realtime 구독을 시작한다

#### Scenario: 이전 방의 늦은 응답
- **WHEN** 방 전환 후 이전 방의 조회 응답이나 실시간 이벤트가 늦게 도착한다
- **THEN** 시스템은 해당 결과를 현재 방 목록에 합치지 않는다

#### Scenario: 비참여자의 그룹방 접근
- **WHEN** 인증 사용자가 참여하지 않은 그룹방의 멤버 또는 메시지에 접근한다
- **THEN** 데이터베이스는 조회와 전송을 거부한다

### Requirement: 기존 공용방 호환성
시스템은 migration 후에도 기존 공용방과 저장된 텍스트 메시지를 MUST 보존해야 한다.

#### Scenario: 기존 사용자의 업데이트 후 접속
- **WHEN** 기존 사용자가 다중 채팅방 버전으로 앱을 연다
- **THEN** 시스템은 기존 공용방 참여와 메시지 내역을 그대로 표시한다
