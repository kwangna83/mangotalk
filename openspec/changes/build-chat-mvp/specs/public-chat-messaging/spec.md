## ADDED Requirements

### Requirement: 공용 채팅방 접근
시스템은 인증된 사용자가 단일 공용 채팅방에 참여하고, 참여 기록을 기반으로 채팅 데이터에 접근할 수 있게 MUST 해야 한다.

#### Scenario: 최초 채팅방 참여
- **WHEN** 인증된 사용자가 처음으로 공용 채팅 화면에 진입한다
- **THEN** 시스템은 해당 사용자의 공용 채팅방 참여 기록을 생성하고 메시지 접근을 허용한다

#### Scenario: 인증되지 않은 접근 차단
- **WHEN** 유효한 인증 세션이 없는 클라이언트가 채팅방 데이터에 접근한다
- **THEN** 데이터베이스는 메시지 조회와 전송 요청을 거부한다

### Requirement: 메시지 내역 조회
시스템은 공용 채팅방의 저장된 메시지를 서버 생성 시각과 ID를 기준으로 일관되게 정렬하여 페이지 단위로 MUST 제공해야 한다.

#### Scenario: 채팅방 최초 진입
- **WHEN** 참여자가 공용 채팅방을 연다
- **THEN** 시스템은 최근 메시지를 최대 50개 조회하여 오래된 것부터 새로운 것 순서로 표시한다

#### Scenario: 이전 메시지 추가 조회
- **WHEN** 참여자가 현재 목록의 가장 오래된 메시지보다 이전 내역을 요청한다
- **THEN** 시스템은 중복 없이 그 이전 메시지를 최대 50개 추가한다

#### Scenario: 메시지가 없는 채팅방
- **WHEN** 저장된 메시지가 없는 공용 채팅방을 연다
- **THEN** 시스템은 오류 대신 빈 상태 안내를 표시한다

### Requirement: 텍스트 메시지 전송
시스템은 참여자가 앞뒤 공백을 제거한 1자 이상 2,000자 이하의 텍스트 메시지를 MUST 전송할 수 있게 해야 한다.

#### Scenario: 유효한 메시지 전송
- **WHEN** 참여자가 유효한 텍스트를 전송한다
- **THEN** 시스템은 서버 시각, 발신자 ID, 채팅방 ID와 함께 메시지를 한 번 저장하고 전송 성공 상태를 표시한다

#### Scenario: 빈 메시지 거부
- **WHEN** 참여자가 비어 있거나 공백만 있는 메시지를 전송하려 한다
- **THEN** 시스템은 데이터베이스 요청을 보내지 않고 전송을 거부한다

#### Scenario: 너무 긴 메시지 거부
- **WHEN** 정리된 메시지가 2,000자를 초과한다
- **THEN** 시스템은 길이 제한을 표시하고 전송을 거부한다

### Requirement: 중복 전송 방지
시스템은 각 전송 시도에 클라이언트 메시지 ID를 부여하고 발신자별로 동일 ID가 한 번만 저장되도록 MUST 보장해야 한다.

#### Scenario: 응답 유실 후 같은 메시지 재시도
- **WHEN** 저장 응답을 받지 못한 클라이언트가 동일한 클라이언트 메시지 ID로 재시도한다
- **THEN** 시스템은 동일 메시지를 추가로 생성하지 않고 기존 저장 결과를 반환한다

### Requirement: 실시간 메시지 수신
시스템은 공용 채팅방에 저장된 새 메시지를 연결된 참여자에게 실시간으로 MUST 전달해야 한다.

#### Scenario: 다른 사용자의 새 메시지 수신
- **WHEN** 다른 참여자가 공용 채팅방에 메시지를 저장한다
- **THEN** 연결된 사용자의 목록에 해당 메시지가 페이지 새로고침 없이 한 번 표시된다

#### Scenario: 자신의 메시지 실시간 이벤트 수신
- **WHEN** 로컬 전송 결과와 동일한 메시지의 실시간 이벤트가 도착한다
- **THEN** 시스템은 메시지 ID를 기준으로 합쳐서 한 번만 표시한다

#### Scenario: 실시간 연결 복구
- **WHEN** 네트워크 중단 후 실시간 연결이 복구된다
- **THEN** 시스템은 마지막으로 확인한 메시지 이후 내역을 조회하여 누락된 메시지를 보완한다

### Requirement: 전송 상태와 재시도
시스템은 사용자가 보낸 메시지의 전송 중, 성공 및 실패 상태를 MUST 구분해 표시해야 한다.

#### Scenario: 메시지 전송 실패
- **WHEN** 메시지 저장 요청이 네트워크 또는 서버 오류로 실패한다
- **THEN** 시스템은 입력 내용을 유지한 실패 상태와 동일 클라이언트 메시지 ID를 사용하는 재시도 동작을 표시한다

### Requirement: 채팅 데이터 접근 제어
시스템은 채팅방 참여자만 메시지를 읽고 쓸 수 있으며 발신자 ID를 위조할 수 없도록 RLS 정책으로 MUST 제한해야 한다.

#### Scenario: 비참여자의 메시지 조회
- **WHEN** 인증 사용자가 참여하지 않은 채팅방의 메시지를 조회한다
- **THEN** 데이터베이스는 메시지를 반환하지 않는다

#### Scenario: 발신자 ID 위조
- **WHEN** 참여자가 다른 사용자 ID를 발신자로 지정하여 메시지를 저장하려 한다
- **THEN** 데이터베이스는 해당 요청을 거부한다
### Requirement: Material 3 Expressive 채팅 표현
시스템은 MangoTalk의 Mango Orange, Leaf Green 및 Purple 의미 색상과 큰 라운드 형태를 사용하는 Material 3 Expressive Community 시각 언어로 채팅 화면을 MUST 표시해야 한다.

#### Scenario: 공용 채팅 화면 표시
- **WHEN** 참여자가 공용 채팅 화면을 연다
- **THEN** 시스템은 표현적인 Mango 헤더, 사용자 아바타, 구분되는 수신·발신 버블, capsule 입력창과 강조된 전송 버튼을 표시한다

#### Scenario: 메시지 콘텐츠 가독성
- **WHEN** 메시지와 장식 요소가 같은 화면에 표시된다
- **THEN** 장식 요소는 메시지 텍스트와 겹치지 않고 본문은 WCAG AA 수준의 색상 대비를 유지한다

### Requirement: 채팅 UI 접근성
시스템은 확대된 텍스트, 화면 읽기 및 모션 감소 설정에서도 핵심 채팅 흐름을 MUST 사용할 수 있게 해야 한다.

#### Scenario: 확대된 시스템 글꼴
- **WHEN** 사용자가 시스템 글꼴 크기를 확대한다
- **THEN** 메시지, 닉네임, 시간과 입력창은 잘리지 않고 읽을 수 있으며 전송 동작에 접근할 수 있다

#### Scenario: 모션 감소 설정
- **WHEN** 사용자가 Reduce Motion을 활성화한다
- **THEN** 시스템은 이동과 크기 변화 애니메이션을 fade 또는 즉시 전환으로 대체한다
