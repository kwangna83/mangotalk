## ADDED Requirements

### Requirement: 알림 권한과 구독
시스템은 사용자의 명시적 동의 후에만 푸시 알림 권한을 요청하고 현재 앱 설치의 FCM 토큰을 인증 사용자에게 MUST 연결해야 한다.

#### Scenario: 사용자가 알림을 허용
- **WHEN** 인증 사용자가 알림 받기를 선택하고 브라우저 또는 운영체제 권한을 허용한다
- **THEN** 시스템은 FCM 토큰을 발급받아 현재 사용자와 기기 설치에 연결하고 알림 활성 상태를 표시한다

#### Scenario: 사용자가 알림을 거부
- **WHEN** 사용자가 알림 권한 요청을 거부한다
- **THEN** 시스템은 채팅 이용을 계속 허용하고 권한 상태와 브라우저 또는 운영체제 설정에서 다시 허용하는 방법을 안내한다

### Requirement: 기기 토큰 수명주기
시스템은 사용자별 복수 기기 토큰을 안전하게 관리하고 토큰 갱신·로그아웃·발송 실패에 따라 등록 상태를 MUST 최신으로 유지해야 한다.

#### Scenario: FCM 토큰 갱신
- **WHEN** FCM이 현재 앱 설치에 새 토큰을 발급한다
- **THEN** 시스템은 기존 설치의 토큰을 새 값으로 교체하여 중복 등록을 만들지 않는다

#### Scenario: 로그아웃
- **WHEN** 사용자가 현재 앱 설치에서 로그아웃한다
- **THEN** 시스템은 해당 설치의 토큰을 현재 사용자에게 더 이상 발송되지 않도록 제거하거나 비활성화한다

#### Scenario: 만료 토큰
- **WHEN** FCM이 등록 토큰이 유효하지 않다고 응답한다
- **THEN** 시스템은 해당 토큰을 비활성화하여 이후 발송 대상에서 제외한다

### Requirement: 새 메시지 푸시 전달
시스템은 저장된 새 메시지를 발신자를 제외한 대상 채팅방 참여자의 활성 기기로 MUST 전달해야 한다.

#### Scenario: 다른 사용자의 새 메시지
- **WHEN** 사용자가 참여 중인 채팅방에 다른 사용자의 메시지가 저장된다
- **THEN** 시스템은 활성 토큰에 발신자 닉네임, 안전하게 제한된 메시지 미리보기와 채팅방 식별자를 포함한 알림을 전송한다

#### Scenario: 발신자 자신의 기기
- **WHEN** 메시지 발신자의 기기 토큰이 발송 후보에 포함된다
- **THEN** 시스템은 해당 사용자 소유 토큰을 발송 대상에서 제외한다

#### Scenario: 중복 webhook 전달
- **WHEN** 동일한 메시지 INSERT 이벤트가 두 번 이상 처리된다
- **THEN** 시스템은 메시지와 토큰 조합별로 알림을 한 번만 발송하거나 동일한 멱등 결과로 처리한다

### Requirement: 앱 상태별 알림 동작
시스템은 앱 상태와 사용자의 현재 위치를 고려하여 불필요한 시스템 알림을 줄이면서 새 메시지를 MUST 알릴 수 있어야 한다.

#### Scenario: 앱이 백그라운드이거나 닫힘
- **WHEN** 알림이 허용된 앱이 백그라운드 또는 종료 상태에서 새 메시지를 수신한다
- **THEN** 운영체제 또는 브라우저는 시스템 알림을 표시한다

#### Scenario: 대상 채팅방을 보고 있음
- **WHEN** 사용자가 포그라운드에서 알림 대상 채팅방을 보고 있다
- **THEN** 시스템은 중복 시스템 알림 대신 기존 실시간 채팅 UI로 메시지를 표시한다

### Requirement: 알림 선택 라우팅
시스템은 사용자가 알림을 선택하면 인증 상태를 복원한 뒤 대상 채팅방으로 MUST 이동해야 한다.

#### Scenario: 유효한 세션으로 알림 선택
- **WHEN** 유효한 세션을 가진 사용자가 메시지 알림을 선택한다
- **THEN** 앱은 알림 payload의 채팅방을 열고 관련 메시지 내역을 표시한다

#### Scenario: 세션이 유효하지 않음
- **WHEN** 세션이 없거나 복원할 수 없는 상태에서 사용자가 알림을 선택한다
- **THEN** 앱은 인증 흐름을 먼저 표시하고 권한이 확인된 경우에만 대상 채팅방으로 이동한다

### Requirement: 알림 데이터와 자격증명 보안
시스템은 Firebase 서버 자격증명을 신뢰할 수 있는 서버 환경에서만 사용하고 알림 데이터 접근을 인증 사용자와 서버 역할로 MUST 제한해야 한다.

#### Scenario: 클라이언트 번들 검사
- **WHEN** Web 또는 모바일 릴리스 번들을 검사한다
- **THEN** Firebase 서비스 계정 private key와 Supabase secret 또는 service-role key가 포함되지 않는다

#### Scenario: 다른 사용자 토큰 접근
- **WHEN** 인증 사용자가 다른 사용자의 기기 토큰을 조회하거나 변경하려 한다
- **THEN** 데이터베이스 RLS는 요청을 거부한다
